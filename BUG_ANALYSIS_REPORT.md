# 功能Bug分析报告

## 概述
本报告针对预计日活一万的正式上线网站进行功能bug分析。重点关注影响用户体验、数据一致性和系统稳定性的问题。

---

## 🔴 严重Bug（必须修复）

### 1. **注册编号并发竞态条件**
**位置**: `backend/core/views.py:464-474`

**问题**: 注册时分配注册编号存在竞态条件。在高并发情况下，多个用户可能获得相同的注册编号。

```python
# 当前代码
max_reg_number = UserProfile.objects.filter(
    registration_number__isnull=False
).aggregate(
    max_num=Max("registration_number")
)["max_num"] or 0

UserProfile.objects.create(
    user=user,
    registration_number=max_reg_number + 1,
)
```

**影响**: 
- 违反唯一性约束，导致注册失败
- 数据不一致

**修复建议**: 使用数据库锁或原子操作：
```python
with transaction.atomic():
    # 使用select_for_update锁定
    max_profile = UserProfile.objects.filter(
        registration_number__isnull=False
    ).order_by('-registration_number').select_for_update().first()
    max_reg_number = max_profile.registration_number if max_profile else 0
    
    UserProfile.objects.create(
        user=user,
        registration_number=max_reg_number + 1,
    )
```

### 2. **Token过期检查缺失**
**位置**: `backend/core/models.py:106-112`

**问题**: `AuthToken.is_expired` 属性存在，但在认证中间件中可能未正确检查。如果token过期，用户仍可能访问受保护资源。

**影响**: 
- 安全漏洞：过期token仍可使用
- 用户体验：突然被登出

**修复建议**: 确保认证中间件检查token过期状态，并在过期时返回401。

---

### 3. **图片上传缺少文件大小限制验证（前端）**
**位置**: `frontend/src/pages/Upload.tsx`

**问题**: 前端未在上传前验证文件大小，用户可能上传超大文件导致：
- 上传失败但用户不知道原因
- 浪费带宽和服务器资源
- 用户体验差

**影响**: 日活一万时，大量失败请求会浪费资源

**修复建议**: 在前端添加文件大小和格式验证，提前提示用户。

---

### 4. **打卡记录并发问题**
**位置**: `backend/core/views.py:1453-1459`

**问题**: 虽然使用了`get_or_create`，但在高并发下，同一用户同一天可能创建多条打卡记录。

**影响**: 数据重复，统计不准确

**修复建议**: 使用数据库唯一约束（已有）配合`get_or_create`，但需要处理`IntegrityError`：
```python
try:
    checkin, created = DailyCheckIn.objects.get_or_create(
        user=user, date=target_date, defaults={"source": source}
    )
except IntegrityError:
    # 并发情况下可能触发，重新获取
    checkin = DailyCheckIn.objects.get(user=user, date=target_date)
    created = False
```

## 🟡 中等严重Bug（建议修复）

### 5. **Gallery页面无限滚动可能导致内存泄漏**
**位置**: `frontend/src/pages/Gallery.tsx:428-463`

**问题**: IntersectionObserver在组件卸载时可能未正确清理，导致内存泄漏。在高日活下会累积。

**影响**: 长期运行后浏览器内存占用增加

**修复建议**: 确保所有observer在useEffect的清理函数中正确disconnect。

---

### 6. **Goals页面批量请求可能导致性能问题**
**位置**: `frontend/src/pages/Goals.tsx:593-713`

**问题**: `refreshCheckInDates`函数在获取目标日期范围时，可能并行请求多个月份的数据。虽然有`MAX_CONCURRENT_REQUESTS=3`限制，但如果目标跨度很大（如90天），仍可能产生大量请求。

**影响**: 
- 服务器负载增加
- 前端可能卡顿
- 网络请求失败率增加

**修复建议**: 
- 增加请求去重机制
- 添加请求取消功能（AbortController）
- 考虑使用缓存策略

---

### 7. **图片压缩失败时错误处理不完善**
**位置**: `backend/core/serializers.py:200-231`

**问题**: 图片压缩失败时抛出ValidationError，但可能包含敏感信息（如PIL的具体错误）。在高并发下，某些恶意文件可能导致服务器资源耗尽。

**影响**: 
- 可能泄露内部错误信息
- 恶意文件可能导致DoS

**修复建议**: 
- 统一错误消息，不暴露内部细节
- 添加超时机制
- 限制图片处理时间

---

### 8. **时区处理不一致**
**位置**: 多处代码

**问题**: 虽然大部分代码使用上海时区，但某些地方可能仍使用本地时区，导致日期计算错误。

**影响**: 打卡日期、统计日期可能不准确

**修复建议**: 统一使用`get_today_shanghai()`函数，确保所有日期计算都基于上海时区。

---

### 9. **前端缓存可能导致数据不一致**
**位置**: `frontend/src/pages/Goals.tsx:323-393`

**问题**: 使用sessionStorage缓存长期目标和短期目标，但缓存失效时间固定为5分钟。如果用户在多个标签页操作，可能导致数据不一致。

**影响**: 用户看到过期数据

**修复建议**: 
- 使用事件通知机制同步多标签页
- 在关键操作后清除缓存
- 考虑使用更短的缓存时间

---

### 10. **Gallery筛选器重置时滚动位置恢复可能失败**
**位置**: `frontend/src/pages/Gallery.tsx:409-426`

**问题**: 筛选器改变时尝试恢复滚动位置，但在某些情况下（如DOM未完全渲染），恢复可能失败或导致页面跳动。

**影响**: 用户体验差

**修复建议**: 
- 使用更可靠的滚动恢复机制
- 添加延迟确保DOM已渲染
- 考虑使用scroll restoration API

---

## 🟢 轻微Bug（可选修复）

### 11. **错误消息不够友好**
**位置**: 多处

**问题**: 某些错误消息对用户不够友好，如"验证码发送失败，请稍后重试"没有提供更多信息。

**影响**: 用户不知道具体问题，难以自行解决

**修复建议**: 提供更详细的错误提示，但不要暴露敏感信息。

---

### 12. **缺少请求重试机制**
**位置**: `frontend/src/services/api.ts`

**问题**: API请求失败时没有自动重试机制，网络波动可能导致操作失败。

**影响**: 用户体验差，特别是在网络不稳定的情况下

**修复建议**: 对非幂等操作添加重试机制（如GET请求）。

---

### 13. **图片懒加载可能重复加载**
**位置**: `frontend/src/pages/Gallery.tsx:465-555`

**问题**: IntersectionObserver的依赖项可能导致observer频繁重建，某些图片可能被重复标记为需要加载。

**影响**: 轻微的性能影响

**修复建议**: 优化依赖项，减少observer重建次数。

---

### 14. **套图切换时索引可能越界**
**位置**: `frontend/src/pages/Gallery.tsx:724-763`

**问题**: 在套图内切换时，虽然有一些边界检查，但在某些边缘情况下（如作品被删除），索引可能仍然越界。

**影响**: 可能导致页面崩溃或显示错误

**修复建议**: 添加更严格的边界检查和错误处理。

---

### 15. **验证码限流可能被绕过**
**位置**: `backend/core/views.py:284-335`

**问题**: 虽然有限流机制，但仅基于IP和邮箱。如果攻击者使用多个IP或邮箱，仍可能绕过限流。

**影响**: 可能被恶意利用发送大量验证码

**修复建议**: 
- 考虑添加更严格的限流策略
- 添加验证码（如reCAPTCHA）
- 监控异常行为

---

## 📊 性能优化建议

### 16. **数据库查询优化**
- Gallery页面加载所有作品，没有分页。当日活一万时，如果用户作品很多，可能导致：
  - 初始加载慢
  - 内存占用高
  - 网络传输量大

**建议**: 实现分页或虚拟滚动

---

### 17. **图片加载优化**
- 虽然实现了懒加载，但初始加载20张图片可能仍然过多

**建议**: 
- 根据设备性能动态调整初始加载数量
- 使用WebP格式（已有）
- 考虑使用CDN

---

### 18. **API响应缓存**
- 某些API（如成就列表）可能被频繁调用，但没有缓存

**建议**: 对不经常变化的数据添加缓存层

---

## 🔒 安全性建议

### 19. **CORS配置**
- 虽然配置了CORS，但需要确保生产环境配置正确

**建议**: 检查生产环境的CORS设置，确保只允许信任的域名

---

### 20. **敏感信息泄露**
- 某些错误消息可能包含敏感信息（如文件路径）

**建议**: 在生产环境统一错误消息，不暴露内部细节
