**审计日期**: 2024年
**审计范围**: 后端Django代码、前端React代码、配置和部署设置
**预期日活**: 10,000用户

## 🔴 严重风险（必须立即修复）

### 1. 安全性问题

#### 1.1 SECRET_KEY 默认值不安全
**位置**: `backend/config/settings.py:31-34`
```python
SECRET_KEY = os.getenv(
    "DJANGO_SECRET_KEY",
    "django-insecure-cek3ua7xe03xkv)eoj&(f30fc8=_^#&l9slh-)vl85$q(qr_o1",
)
```
**风险**: 如果环境变量未设置，会使用硬编码的不安全密钥
**影响**: 攻击者可以伪造session、签名等，完全控制应用
**修复**: 
- 生产环境必须设置 `DJANGO_SECRET_KEY` 环境变量
- 移除默认值，如果未设置则抛出异常
- 使用强随机生成的密钥（至少50字符）

#### 1.2 SMTP密码硬编码
**位置**: `backend/config/settings.py:354`
```python
EMAIL_HOST_PASSWORD = os.getenv("SMTP_PASSWORD", "RLTFvFP3ReVTGhh8")
```
**风险**: 密码泄露在代码中，可能被提交到版本控制
**影响**: 邮件服务可能被滥用，发送垃圾邮件
**修复**: 
- 立即更换密码
- 必须从环境变量读取，无默认值
- 检查git历史，确保未提交到仓库

#### 1.3 DEBUG模式可能在生产环境开启
**位置**: `backend/config/settings.py:37`
```python
DEBUG = os.getenv("DJANGO_DEBUG", "True").lower() == "true"
```
**风险**: 默认开启DEBUG，会暴露敏感信息
**影响**: 错误页面会显示完整堆栈跟踪、SQL查询、环境变量等
**修复**: 
- 默认值改为 `"False"`
- 生产环境必须明确设置 `DJANGO_DEBUG=false`
- 添加检查：如果DEBUG=True且不在本地环境，抛出异常

#### 1.4 CORS配置过于宽松
**位置**: `backend/config/settings.py:272-298`
```python
if not CORS_ALLOWED_ORIGINS:
    CORS_ALLOWED_ORIGINS = [
        "http://localhost:3000",
        "http://localhost:5173",
        ...
    ]
if DEBUG:
    CORS_ALLOW_ALL_ORIGINS = True
```
**风险**: DEBUG模式下允许所有来源，可能被CSRF攻击
**影响**: 恶意网站可以跨域访问API
**修复**: 
- 生产环境必须明确配置 `DJANGO_CORS_ALLOWED_ORIGINS`
- 移除DEBUG模式下的 `CORS_ALLOW_ALL_ORIGINS = True`
- 使用白名单机制

#### 1.5 文件上传验证不足
**位置**: `backend/core/serializers.py:88-100`
**风险**: 
- 虽然有MIME类型检查，但缺少文件大小限制验证
- 缺少文件内容验证（可能上传伪装的文件）
- 没有速率限制，可能被DoS攻击
**影响**: 
- 服务器存储空间被耗尽
- 可能上传恶意文件
- 大文件上传导致服务器资源耗尽
**修复**: 
- 在serializer中添加文件大小验证（当前settings中只有10MB限制）
- 使用PIL验证图片文件头，不只是扩展名
- 添加上传速率限制（每个用户每小时最多X次）
- 考虑添加病毒扫描（可选）

#### 1.6 Token无过期时间
**位置**: `backend/core/models.py:75-96`
**风险**: AuthToken永不过期，一旦泄露永久有效
**影响**: 如果token泄露，攻击者可以长期访问账户
**修复**: 
- 添加token过期时间（建议30天）
- 添加token刷新机制
- 记录token使用日志，异常使用自动失效

### 2. 性能问题

#### 2.1 数据库查询N+1问题
**位置**: 多处，例如：
- `backend/core/views.py:1384` - 循环中查询数据库
- `backend/core/views.py:1267-1274` - 循环处理上传记录
**风险**: 在高并发下会导致数据库连接耗尽
**影响**: 响应时间变慢，可能导致服务不可用
**修复**: 
- 使用 `select_related()` 和 `prefetch_related()` 优化关联查询
- 批量处理数据，减少循环中的查询
- 使用 `values_list()` 时确保只获取需要的字段

#### 2.2 缺少数据库连接池配置
**位置**: `backend/config/settings.py:112,131,148`
**风险**: 默认连接池可能不够
**影响**: 高并发时可能耗尽数据库连接
**修复**: 
- 配置合适的 `CONN_MAX_AGE`（当前60秒可能不够）
- 考虑使用连接池（如pgBouncer for PostgreSQL）
- 监控数据库连接数

#### 2.3 缓存策略不足
**位置**: 整个项目
**风险**: 只有少量 `cache_page`，缺少对象级缓存
**影响**: 重复查询相同数据，浪费资源
**修复**: 
- 对频繁访问的数据使用Redis缓存
- 缓存成就列表、任务预设等静态数据
- 使用缓存版本控制，确保数据更新时失效

#### 2.4 邮件发送可能阻塞
**位置**: `backend/core/email_utils.py:21-45`
**风险**: 虽然使用了线程池，但线程池大小固定（4个）
**影响**: 高并发时邮件发送可能排队，影响响应时间
**修复**: 
- 优先使用Celery异步任务队列
- 增加线程池大小或使用动态线程池
- 添加邮件发送失败重试机制

## 🟡 中等风险（建议尽快修复）

### 3. 并发和竞态条件

#### 3.1 验证码发送频率限制可能不够
**位置**: `backend/core/views.py:252-262`
```python
if latest_record and (now - latest_record.created_at).total_seconds() < RESEND_INTERVAL_SECONDS:
```
**风险**: 60秒限制可能不够，且没有IP级别的限制
**影响**: 可能被滥用发送大量邮件，导致邮件服务被限制
**修复**: 
- 添加IP级别的频率限制
- 增加每日发送上限（如每个邮箱每天最多10次）
- 使用Redis实现分布式限流

#### 3.2 打卡操作可能存在竞态条件
**位置**: `backend/core/views.py:1348-1353`
```python
checkin, created = DailyCheckIn.objects.get_or_create(
    user=user, date=target_date, defaults={"source": source}
)
```
**风险**: 虽然使用了 `get_or_create`，但在高并发下仍可能出现重复
**影响**: 可能导致数据不一致
**修复**: 
- 数据库层面已有唯一约束（`unique_daily_checkin_per_user`），这是好的
- 但可以考虑添加分布式锁（Redis）防止重复提交

#### 3.3 用户注册可能存在竞态条件
**位置**: `backend/core/views.py:340-344,375-381`
**风险**: 检查用户是否存在和创建用户之间有时间窗口
**影响**: 可能创建重复用户
**修复**: 
- 数据库层面添加唯一约束（email字段）
- 使用 `transaction.atomic()` 确保原子性（已有）

### 4. 错误处理和日志

#### 4.1 某些错误信息可能泄露敏感信息
**位置**: `backend/core/views.py:296-298`
```python
if settings.DEBUG:
    payload["error"] = _extract_error_message(exc, "邮件发送失败")
```
**风险**: DEBUG模式下可能泄露内部错误信息
**影响**: 攻击者可能获取系统内部信息
**修复**: 
- 生产环境不返回详细错误信息
- 使用统一的错误处理中间件
- 详细错误只记录到日志

#### 4.2 缺少请求超时设置
**位置**: 整个项目
**风险**: 长时间运行的请求可能占用资源
**影响**: 可能导致服务器资源耗尽
**修复**: 
- 配置请求超时（如30秒）
- 对长时间运行的任务使用异步处理

### 5. 数据验证

#### 5.1 邮箱验证码可能被暴力破解
**位置**: `backend/core/views.py:352-373`
**风险**: 6位数字验证码，只有100万种可能
**影响**: 理论上可以暴力破解（虽然有时间限制）
**修复**: 
- 添加尝试次数限制（如5次失败后锁定）
- 考虑增加验证码复杂度或使用更安全的验证方式

#### 5.2 密码强度验证可能不够
**位置**: `backend/core/views.py:335-338`
```python
if len(password) < 8:
    return Response(
        {"detail": "密码长度至少 8 位"}, status=status.HTTP_400_BAD_REQUEST
    )
```
**风险**: 只检查长度，不检查复杂度
**影响**: 用户可能使用弱密码
**修复**: 
- 使用Django的密码验证器（已在settings中配置，但views中未使用）
- 确保密码包含大小写字母、数字、特殊字符

## 📊 性能估算（日活1万）

### 数据库连接需求
- 假设平均每个请求需要100ms数据库时间
- 峰值QPS: 约100-200（假设用户活跃时间集中在2-3小时）
- 需要的数据库连接: 约20-40个
- **建议**: 配置至少50个连接池

### 存储需求
- 假设每个用户每天上传1张图片（平均2MB）
- 每天新增: 10,000 * 2MB = 20GB
- 每月新增: 约600GB
- **建议**: 使用对象存储（TOS），配置生命周期策略

### 缓存需求
- 热点数据（成就列表、任务预设）: 约10-50MB
- 用户session/token: 约100MB（假设1万活跃用户）
- **建议**: 使用Redis，至少512MB内存

### 邮件发送需求
- 注册/重置密码: 假设每天100次
- 验证码有效期10分钟，重发间隔60秒
- **建议**: 当前配置足够，但需要监控邮件服务商限制
